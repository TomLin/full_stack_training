## 1. 已定案協議 (Finalized Agreements)

### 1.1 核心策略決策

#### D-001: 平台開發模式

| 項目         | 內容                                                                                     |
| ------------ | ---------------------------------------------------------------------------------------- |
| **議題**     | 採用「單邊平台」還是「雙邊平台」？                                                       |
| **討論背景** | 雙邊平台（業主+志工）技術含量較高，但工時倍增                                            |
| **決議**     | 採用**「模擬雙邊，實作單邊」**策略                                                       |
| **優先級**   | 🔴 **MUST**                                                                              |
| **理由**     | 架構上設計完整的雙邊角色，但實作重心放在「志工尋找職缺」的路徑，集中火力展示前端互動能力 |
| **影響範圍** | 整體系統架構、頁面規劃、資料結構                                                         |

---

#### D-002: 資料來源策略

| 項目         | 內容                                                                              |
| ------------ | --------------------------------------------------------------------------------- |
| **議題**     | 如何處理資料來源？真實後端還是模擬？                                              |
| **討論背景** | 專案為前端訓練班結業專題，無後端資源                                              |
| **決議**     | 使用**靜態 `jobs.json` 檔案**模擬 API 資料來源                                    |
| **優先級**   | 🔴 **MUST**                                                                       |
| **理由**     | 學員可藉此練習 `fetch` API 與非同步資料處理，後續對接真實後端時僅需替換資料介接層 |
| **影響範圍** | 資料結構設計、前端邏輯實作                                                        |

---

#### D-003: 會員系統必要性

| 項目         | 內容                                                                                                                                      |
| ------------ | ----------------------------------------------------------------------------------------------------------------------------------------- |
| **議題**     | 是否需要登入功能？還是用 localStorage 即可？                                                                                              |
| **討論背景** | 有人提議既然 localStorage 可以儲存偏好，是否可以跳過登入                                                                                  |
| **決議**     | **必須實作完整的會員登入與註冊流程**                                                                                                      |
| **優先級**   | 🔴 **MUST**                                                                                                                               |
| **理由**     | 基於**「可信度檢驗 (Credibility Verification)」**。換宿涉及實體居住安全，非登入的匿名狀態無法建立足夠的信任感，亦不符合此類產品的業界標準 |
| **隱含協議** | - 訪客可瀏覽但不可應徵<br>- 登入後才能收藏、應徵<br>- 會員中心必須存在                                                                    |
| **影響範圍** | 權限控管、Sitemap、UI 狀態設計                                                                                                            |

---

#### D-004: 新手引導 (Onboarding) 強制性

| 項目         | 內容                                                                                                  |
| ------------ | ----------------------------------------------------------------------------------------------------- |
| **議題**     | Onboarding 是選填還是必填？                                                                           |
| **討論背景** | 希望避免使用者反覆填寫資料，並解決首頁「冷啟動」問題                                                  |
| **決議**     | **註冊成功後強制進入 Onboarding Wizard**                                                              |
| **優先級**   | 🔴 **MUST**                                                                                           |
| **理由**     | 確保每位會員都有偏好標籤，以利首頁進行精準推薦，解決冷啟動問題                                        |
| **隱含協議** | - Onboarding 不可跳過<br>- 未完成 Onboarding 不可進入首頁<br>- 偏好設定存入 localStorage 或模擬資料庫 |
| **影響範圍** | 註冊流程、首頁推薦邏輯                                                                                |

---

#### D-005: 媒合流程邊界

| 項目         | 內容                                                                                                           |
| ------------ | -------------------------------------------------------------------------------------------------------------- |
| **議題**     | 媒合過程是否要完整做在系統內？                                                                                 |
| **討論背景** | 完整的即時通訊系統開發成本過高                                                                                 |
| **決議**     | 採用**「站內投遞，站外聯繫」**模式                                                                             |
| **優先級**   | 🔴 **MUST**                                                                                                    |
| **理由**     | - 保留平台數據價值（應徵紀錄）<br>- 避免開發複雜的即時通訊系統<br>- 送出申請後引導雙方透過 Email/Line 私下聯繫 |
| **隱含協議** | - 必須有「應徵」按鈕與確認流程<br>- 必須有「應徵紀錄」頁面<br>- 不做站內聊天功能                               |
| **影響範圍** | 應徵流程 UI、會員中心設計                                                                                      |

### 1.2 資料結構決策

#### D-006: 工作類型資料格式

| 項目       | 內容                                                                                                 |
| ---------- | ---------------------------------------------------------------------------------------------------- |
| **議題**   | 工作類型用單一字串還是陣列？                                                                         |
| **選項**   | A) 單一字串 `"type": "房務"` <br> B) 字串陣列 `"types": ["房務", "小編"]`                            |
| **決議**   | 採用**字串陣列 (Array of Strings)**                                                                  |
| **優先級** | 🔴 **MUST**                                                                                          |
| **理由**   | - 符合現實（工作通常是複合型）<br>- 讓學員練習 `Array.prototype.some()` 或 `filter` 等較進階陣列操作 |

---

#### D-007: 可入住時間格式

| 項目       | 內容                                                                                             |
| ---------- | ------------------------------------------------------------------------------------------------ |
| **議題**   | 可入住時間用精確日期還是月份？                                                                   |
| **選項**   | A) 月份字串 `["2025-08", "2025-09"]` <br> B) 日期區間 `startDate`, `endDate`                     |
| **決議**   | 採用**月份字串陣列**                                                                             |
| **優先級** | 🟡 **SHOULD**                                                                                    |
| **理由**   | - 避免處理複雜的日期區間 (Date Range) 邏輯<br>- 降低新手門檻<br>- 仍能做到「月份篩選」的功能展示 |

---

#### D-008: 結構化詳情欄位

| 項目       | 內容                                   |
| ---------- | -------------------------------------- |
| **議題**   | 職缺詳情要包含哪些結構化欄位？         |
| **決議**   | 必須包含以下結構化欄位（非純文字描述） |
| **優先級** | 🔴 **MUST**                            |

**工作條件欄位：**

- 工作類型 (types[])
- 最短工期 (minWeeks)
- 每日工時 (workHours)
- 休假制度 (daysOff)
- 排班方式 (shift)
- 勞動強度 (laborIntensity)
- 保險投保 (insurance)

**住宿條件欄位：**

- 住宿類型 (roomType)
- 供餐情況 (meals)
- 冷氣電費 (acFee) ⚠️ 重要地雷欄位
- 衛浴隱私 (bathroom)
- 室友組合 (roommates)
- 額外福利 (benefits[])
- 設施 (amenities[])

**生活公約欄位：**

- 門禁時間 (curfew)
- 訪客規定 (visitor)
- 寵物友善 (petFriendly)
- 菸酒規定 (smoking)

**理由** | 「把資訊陳列清楚，就能大幅降低溝通成本」，避免資訊不對稱造成糾紛 |

### 1.3 篩選與導覽決策

#### D-009: 必要篩選功能

| 項目         | 內容                                                                   |
| ------------ | ---------------------------------------------------------------------- |
| **議題**     | 列表頁必須實作哪些篩選？                                               |
| **決議**     | 至少實作**「地點」與「換宿類型」**兩組篩選器                           |
| **優先級**   | 🔴 **MUST**                                                            |
| **理由**     | 這是媒合平台的核心功能。若無篩選，列表頁僅是靜態展示，無法體現工程邏輯 |
| **隱含協議** | - 篩選器之間為 AND 關係<br>- 同一篩選器內的選項為 OR 關係              |

---

#### D-010: 表單送出行為

| 項目       | 內容                                                                  |
| ---------- | --------------------------------------------------------------------- |
| **議題**   | 聯絡表單送出後如何回饋？                                              |
| **選項**   | A) JavaScript Alert <br> B) 內容替換為感謝文字 <br> C) 重新導向回首頁 |
| **決議**   | 顯示**成功彈窗 (Alert/Modal)**並清空欄位，不實作 SMTP 寄信            |
| **優先級** | 🟡 **SHOULD**                                                         |
| **理由**   | 純前端環境無法處理 SMTP 寄信。此決策足以展示「表單驗證」與 DOM 操作   |

### 1.4 評論系統決策

#### D-011: 評論結構化標籤

| 項目         | 內容                                                                                 |
| ------------ | ------------------------------------------------------------------------------------ |
| **議題**     | 評論只有文字還是要有結構化選項？                                                     |
| **決議**     | 採用**「結構化勾選標籤 + 文字心得」**雙軌設計                                        |
| **優先級**   | 🟡 **SHOULD**                                                                        |

| **標籤類別** | 符合度（工作與描述相符、無隱藏收費）、設施（Wifi 穩、冷氣涼）、氛圍（社交型/獨立型） |

| ------------ | -------------------------------------------------- |

## 2. 待觀察決策 (Decisions Under Review)

> ⚠️ 以下決策已暫時定案，但明確標記為「可推翻」，後續應重新檢視：

### D-014: 列表頁導覽模式 (地點優先)

| 項目             | 內容                                                                                       |
| ---------------- | ------------------------------------------------------------------------------------------ |
| **議題**         | 列表頁的主要導覽邏輯是什麼？                                                               |
| **暫定方案**     | 採用**「地點優先 (Location-First)」**介面，類似 Uber Eats 的「置頂地點切換」模式           |
| **優先級**       | 🟡 **SHOULD**                                                                              |
| **理由**         | 換宿使用者通常先決定「去哪裡」，地點應作為最高層級的篩選                                   |
| **待觀察點**     | - 假設使用者都已有明確目的地<br>- 如果測試後發現使用者更喜歡無目的瀏覽，此設計可於日後推翻 |
| **可能替代方案** | - 純探索式瀏覽（無預設地點）<br>- 地圖優先模式                                             |
| **重新檢視時機** | 使用者測試後、MVP 上線後                                                                   |

---

### D-015: 驗證等級設計

| 項目             | 內容                                                   |
| ---------------- | ------------------------------------------------------ |
| **議題**         | 會員驗證要分幾個等級？                                 |
| **暫定方案**     | 三級制（L0 訪客、L1 Email 驗證、L2 實名認證）          |
| **優先級**       | 🟢 **COULD**                                           |
| **待觀察點**     | - 實名認證在純前端環境難以真正驗證<br>- 是否過度設計？ |
| **重新檢視時機** | 後端開發階段                                           |

---

### D-016: 工作類型分類

| 項目             | 內容                                                                                                                               |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| **議題**         | 工作類型要有哪些具體分類？                                                                                                         |
| **暫定方案**     | 核心勞務類（房務、櫃台、廚房、農務、園藝）+ 專業技能類（小編、攝影、美術、網站、瑜伽教學）+ 總務活動類（工務、活動、寵物、小管家） |
| **優先級**       | 🟡 **SHOULD**                                                                                                                      |
| **待觀察點**     | - 分類是否過多或過少？<br>- 是否需要大分類+小分類的階層？                                                                          |
| **重新檢視時機** | 假資料建立時、使用者回饋後                                                                                                         |

## 4. 討論過程紀錄 (Discussion Timeline)

## MoSCoW 總覽

### 🔴 MUST (必須有)

| 項目       | 決策/協議                    |
| ---------- | ---------------------------- |
| 平台模式   | 單邊核心（志工端優先）       |
| 資料來源   | 靜態 JSON 模擬               |
| 會員系統   | 必須有登入/註冊              |
| Onboarding | 強制執行                     |
| 技術堆疊   | HTML/CSS/JS/jQuery/Bootstrap |
| RWD        | 桌機+手機版                  |
| 核心頁面   | 首頁、列表頁、詳情頁         |
| 篩選功能   | 地點+類型篩選                |
| 應徵流程   | 站內投遞（有應徵紀錄）       |
| 資料結構   | 工作類型用陣列               |

### 🟡 SHOULD (應該有)

| 項目            | 決策/協議                      |
| --------------- | ------------------------------ |
| 導覽模式        | 地點優先                       |
| 評論系統        | 結構化標籤 + 巢狀回覆          |
| 詳情頁          | 結構化欄位（工時/供餐/冷氣等） |
| Onboarding 形式 | Modal 強制完成                 |
| 空狀態設計      | 搜尋無結果要有引導             |
| 麵包屑導航      | 層級導引                       |
| 表單反饋        | Alert/Modal 成功訊息           |

### 🟢 COULD (可以有)

| 項目     | 決策/協議        |
| -------- | ---------------- |
| 多語系   | 繁中/EN 切換     |
| 深色模式 | 夜間瀏覽         |
| 地圖模式 | 視覺化地理搜尋   |
| 最近瀏覽 | 歷史紀錄         |
| 社群分享 | Line/FB 分享按鈕 |
| 檢舉功能 | 社群監督         |
| 驗證等級 | L0/L1/L2 三級制  |
| 通知中心 | 系統通知         |
| 骨架屏   | 載入效果         |
| 換宿故事 | 軟性內容策展     |

### ⚪ WON'T (不會有 / 延後)

| 項目           | 理由                       |
| -------------- | -------------------------- |
| 業主後台       | 專注志工端，後續課程再做   |
| 即時通訊       | 開發成本過高，改用站外聯繫 |
| SMTP 寄信      | 純前端無法處理             |
| 真實金流       | 非專案範圍                 |
| 真實後端資料庫 | 用 localStorage 模擬       |
| 真實身分驗證   | UI 示意即可                |

## 5. 新增決策 (2024-12-07)

### D-018: 品牌名稱統一

| 項目         | 內容                                                       |
| ------------ | ---------------------------------------------------------- |
| **議題**     | 專案品牌名稱應該使用 Stayventure 還是 Builder 101？        |
| **討論背景** | 原始規劃使用 Stayventure，但業主需求文件指定 Builder 101   |
| **決議**     | 採用 **Builder 101** 作為正式品牌名稱                      |
| **優先級**   | 🔴 **MUST**                                                |
| **理由**     | 配合業主需求文件 (requirement.md) 的明確指定               |
| **影響範圍** | 所有文件、Vue 組件、HTML 標題、meta 描述、localStorage key |

---

### D-019: 導覽列命名簡化

| 項目         | 內容                                                  |
| ------------ | ----------------------------------------------------- |
| **議題**     | 導覽列項目應使用原始冗長名稱還是簡化版？              |
| **選項**     | A) 尋求項目/民宿主 vs B) 探索項目                     |
| **決議**     | 採用**簡化版命名**                                    |
| **優先級**   | 🟡 **SHOULD**                                         |
| **理由**     | 簡潔清晰，符合現代 UI 設計原則，且不影響使用者理解    |
| **具體變更** | - 尋求項目/民宿主 → 探索項目<br>- 聯繫我們 → 聯絡我們 |

### D-021: Hero Carousel 圖片尺寸

| 項目       | 內容                                                          |
| ---------- | ------------------------------------------------------------- |
| **議題**   | Hero Carousel 圖片應使用多大尺寸？                            |
| **決議**   | 建議使用 **1920 x 600 px** (16:5 比例)                        |

### D-022: Toast 訊息回饋

| 項目         | 內容                                                     |
| ------------ | -------------------------------------------------------- |
| **議題**     | 表單送出和互動操作應如何提供使用者回饋？                 |
| **決議**     | 使用 **Bootstrap Toast** 組件顯示操作結果                |
