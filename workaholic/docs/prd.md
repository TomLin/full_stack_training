## 1. 專案背景與目標 (Project Overview)

### 1.1 專案性質

-   **專案類型：** 前端工程師訓練班結業專題
-   **核心目標：** 打造一個具備「高可信度」與「完整會員體驗」的打工換宿媒合平台
-   **技術邊界：** 純前端實作 (HTML/CSS/JS/jQuery/Bootstrap)，使用 `localStorage` 模擬資料庫與登入狀態，使用 JSON 檔案模擬 API 資料

### 1.2 產品願景

Builder 101 旨在解決「志工 (Builder)」與「民宿/業主 (Host)」之間的信任與媒合問題。不同於一般的資訊佈告欄，Builder 101 定位為一個**「高可信度的媒合生態系」**。系統設計必須建立在「身份識別」與「資訊透明」的基礎上。

### 1.3 開發策略

| 策略項目         | 說明                                                                                             |
| ---------------- | ------------------------------------------------------------------------------------------------ |
| **單邊核心**     | 鎖定「志工 (Builder) 尋找職缺」的使用者路徑 (Happy Path)，暫不實作業主後台                       |
| **模擬後端**     | 使用靜態 `jobs.json` 檔案模擬 API 資料來源                                                       |
| **技術展示優先** | 犧牲部分商業邏輯（如真實寄信、真實金流），換取實作「前端篩選」、「動態路由」與「狀態管理」的機會 |

## 8. 業務流程敘事 (Business Process Narrative)

### 第一階段：初識與轉化 (Acquisition & Cold Start)

當一位對打工換宿感興趣的使用者（Builder）首次造訪 Builder 101 時，系統會先以「訪客模式」接待他。此時，首頁展示的是全站通用的熱門職缺，旨在快速吸引他的目光。

為了提供精準的媒合，平台的核心業務目標是盡快將這位訪客轉化為註冊會員。當 Builder 被某個職缺吸引，試圖點擊「收藏」或「應徵」時，系統會介入並引導他完成註冊。

註冊成功後，Builder 會進入一個**強制性的新手引導流程 (Mandatory Onboarding)**。系統會詢問兩個關鍵問題：「你想體驗什麼樣的生活？」以及「你對住宿有什麼偏好？」

一旦完成作答，系統的演算法隨即啟動。當他再次回到首頁時，版面已煥然一新，變成了根據他偏好生成的「個人化推薦清單」。至此，系統解決了「冷啟動」問題，Builder 正式成為平台生態系的一員。

### 第二階段：探索與信任建立 (Discovery & Evaluation)

進入媒合階段，Builder 的行為模式轉為以「地點」為核心。系統理解換宿者通常先決定「去哪裡」，因此提供了**以地點為優先的導覽體驗**。Builder 可以鎖定特定地區，並透過「智慧篩選器」過濾出符合條件的職缺。

在瀏覽職缺詳情時，業務流程的重點在於**「建立信任」**。為了避免資訊不對稱造成的糾紛，系統強制業主必須結構化地揭露關鍵資訊，如「每日工時強度」、「是否需自付電費」以及「具體的住宿房型」。

此外，Builder 還能查看過往志工留下的評價。這些評價不僅有文字，還包含「Wifi 訊號穩」、「供餐好吃」等結構化標籤，幫助 Builder 快速判斷職缺的真實性與品質。

### 第三階段：承諾與遞送 (Commitment & Transaction)

當 Builder 決定應徵時，系統首先會進行權限檢核，確保 Builder 已登入且帳號狀態正常。

系統設有**「防呆與防擾機制」**：

1. Builder 必須填寫一份不少於 50 字的自我推薦信
2. 系統會檢查過去 7 天內是否已應徵過同一份工作

當 Builder 按下「確認送出」後，系統會：

1. 將申請單封裝，標記狀態為 [已送出]
2. 將 Builder 的個人檔案與履歷快照存入業主的收件匣
3. 明確告知：「業主將透過 Email 或 Line 進行後續聯繫」

這就是**「站內投遞，站外聯繫」**的邊界，既保留了平台的數據價值，也簡化了即時通訊的開發成本。

### 第四階段：迴圈與資產累積 (Retention & Feedback)

應徵送出後，Builder 可以隨時回到「會員中心」追蹤所有申請的進度。系統會模擬真實世界的時間差，將狀態從 [已送出] 更新為 [業主已讀] 甚至 [邀請面試]。

當換宿結束後，業務流程會引導 Builder 回到平台貢獻內容，撰寫評論。系統採用「巢狀回覆」機制，允許業主對評論進行回覆。

這些累積的數據——應徵紀錄、收藏列表、撰寫的評價——共同構成了 Builder 在平台上的**「信用資產」**，形成一個正向的業務迴圈。

## 9. 業務規則與限制 (Business Rules)

| 規則               | 說明                                                                    |
| ------------------ | ----------------------------------------------------------------------- |
| **重複應徵限制**   | 同一位會員對同一個職缺在 7 天內只能應徵一次                             |
| **資料完整性規則** | 應徵時，自我推薦信不得少於 50 字                                        |
| **隱私保護規則**   | 在雙方未達成媒合意向前，平台不顯示業主的私人聯絡方式                    |
| **下架規則**       | 若職缺可入住月份已過期，系統自動標示為「已額滿/停止招募」並隱藏應徵按鈕 |
